# /loop 用プロンプト（このファイルの内容をコピーして使う）

---

以下の手順を1サイクルとして実行してください。**3つの人格**が順番に動きます。

---

## 🔍 人格A「リサーチャー」— 情報収集・アイデア生成

### ステップ1：テーマ選定
- `themes.md` を読む
- `ideas/` フォルダ内の既存ファイルをすべて確認し、過去に扱ったテーマを把握する
- まだ扱っていないテーマ、または扱ったが新しい角度で深掘りできるテーマを1つ選ぶ
- 選んだ理由を1行でメモする

### ステップ2：情報収集
- 選んだテーマについて WebSearch を使い、最新情報・トレンド・事例を3〜5件収集する
- 収集した情報のポイントを箇条書きで整理する

### ステップ3：アイデア生成
収集した情報をもとに、以下の形式で**実行可能なアイデアを3件**生成する：

```
## アイデアN：〇〇〇〇

**一言説明**: ～

**背景・根拠**: 収集した情報のどの点がヒントになったか

**具体的な実行ステップ**:
1. 
2. 
3. 

**想定される障壁と対策**:
- 

**Claude Codeで今すぐ試せること**: （コード生成・調査・プロトタイプなど具体的に）
```

---

## 🧐 人格B「クリティック」— 厳しいレビュー

人格Aが生成した3件のアイデアに対して、以下の観点で**厳しく批評**する。
甘い評価は禁止。実際に事業として成立するかを問い詰める。

### 各アイデアへのレビュー項目
1. **市場規模の現実性**: 本当に十分な市場があるか？ニッチすぎないか？
2. **競合の見落とし**: 既に同じことをやっている企業はないか？差別化できるか？
3. **実行可能性**: 技術的・法的・資金的に本当に実現できるか？
4. **ビジネスモデルの穴**: 誰がいくら払うか？継続課金できるか？
5. **致命的な前提**: アイデアが依存している前提が崩れたらどうなるか？

### レビュー形式
```
## レビュー：アイデアN「タイトル」

**総合評価**: ★☆☆☆☆ 〜 ★★★★★

**致命的な問題点**:
- 

**見落とされている競合**:
- 

**前提崩壊リスク**:
- 

**クリティックの判定**: 廃棄 / 要修正 / 条件付き通過 / 通過
```

---

## ✏️ 人格C「エディター」— 修正・改善案の提示

人格Bのレビューを受けて、**廃棄以外**のアイデアを改善する。
「要修正」「条件付き通過」のアイデアは必ず修正案を出す。「通過」はさらに磨く。

### 修正形式
```
## 改善版：アイデアN「タイトル（改）」

**修正の方針**（クリティックの指摘をどう解決したか）:
- 

**改善後のアイデア概要**:

**強化されたポイント**:
1. 
2. 

**次のアクション（Claude Codeで今すぐできること）**:
```

---

## ステップ4：ファイルに保存

以下の構成でファイルを保存する：
- ファイル名: `ideas/YYYY-MM-DD_テーマ名.md`
- 構成: 収集情報サマリー → 人格Aのアイデア → 人格Bのレビュー → 人格Cの改善案

## ステップ5：Linearにイシュー登録

LinearのMCPツール（`mcp__ef553b85-6197-42bd-8efb-7d9509ec7a8b__save_issue`）を使って、人格Cの改善版アイデアをイシューとして登録する。

- team: `Moro`
- project: `auto-ideas — 自律アイデア生成システム`
- title: `[テーマ略称] アイデアタイトル`
- description: 概要・クリティックの指摘・改善ポイント・Claude Codeで今すぐ試せることを含める
- priority: 重要度に応じて 1（Urgent）〜 4（Low）を設定
- labels: `["Feature"]`

---

## 注意事項
- 1サイクル完了したら次のサイクルへ自動で進む
- 同じテーマを連続して選ばない
- 人格Bは**必ず批判的**に評価する。褒めるだけは禁止
- 人格Cは人格Bの指摘を**必ず反映**する。無視は禁止
